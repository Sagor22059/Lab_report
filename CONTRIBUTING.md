# Contributing to Lab Report System

Thank you for your interest in contributing to the Lab Report System! We welcome contributions from the community.

## 🤝 How to Contribute

### Reporting Bugs
1. Check the [Issues](https://github.com/Sagor22059/Lab_report/issues) to ensure the bug hasn't been reported
2. Create a new issue with:
   - Clear title describing the bug
   - Detailed description of the issue
   - Steps to reproduce
   - Expected vs actual behavior
   - Screenshots if applicable

### Suggesting Enhancements
1. Create an issue with the "enhancement" label
2. Clearly describe the feature and its benefits
3. Provide examples of how it would be used

### Code Contributions

#### Getting Started
1. Fork the repository
2. Clone your fork locally
   ```bash
   git clone https://github.com/YOUR_USERNAME/Lab_report.git
   cd Lab_report
   ```
3. Create a new branch
   ```bash
   git checkout -b feature/AmazingFeature
   ```

#### Development Setup
1. Install dependencies
   ```bash
   npm install
   ```
2. Set up `.env` file with your configuration
3. Start the development server
   ```bash
   npm run dev
   ```

#### Making Changes
1. Write clean, readable code
2. Follow the existing code style
3. Add comments for complex logic
4. Test your changes thoroughly

#### Commit Guidelines
- Use meaningful commit messages
- Follow the pattern: `[Type] Brief description`
- Types: feat, fix, docs, style, refactor, test, chore

Examples:
```
feat: Add email notifications for submissions
fix: Correct database connection error handling
docs: Update installation instructions
```

#### Pushing Changes
1. Push to your fork
   ```bash
   git push origin feature/AmazingFeature
   ```
2. Create a Pull Request with:
   - Clear title and description
   - Link to related issues
   - Screenshots if applicable
   - Test results

### Pull Request Process
1. Ensure code follows the project's style
2. Update documentation if needed
3. Add tests for new features
4. Ensure all tests pass
5. Wait for code review

## 📋 Code Style Guidelines

### JavaScript
- Use 4 spaces for indentation
- Use meaningful variable and function names
- Add JSDoc comments for functions
- Avoid console.log in production code

### EJS Templates
- Use consistent indentation
- Add comments for complex logic
- Keep templates lean (logic in routes)

### CSS
- Use CSS variables for colors
- Mobile-first responsive design
- Follow BEM naming convention when applicable

## 🧪 Testing
- Test all changes locally
- Verify on different browsers if possible
- Test both student and teacher workflows
- Check mobile responsiveness

## 📚 Documentation
- Update README if adding features
- Document new endpoints/routes
- Add code comments for complex logic
- Update database schema if modified

## 🚫 Code of Conduct
- Be respectful and inclusive
- Provide constructive feedback
- Help others learn and grow
- Report inappropriate behavior

## ❓ Questions?
- Check existing issues and discussions
- Create a new discussion if needed
- Comment on related issues
